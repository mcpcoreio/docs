## How it works

In **Bearer Token** mode, clients send a signed JWT in the `Authorization: Bearer` header. MCPCore verifies the token's signature using a public key you configure, and rejects tokens that are expired or have an invalid signature.

You are responsible for issuing the tokens from your own backend. This mode is designed for **server-to-server** integrations where tokens are generated programmatically rather than through a user-facing login flow.

```
Your backend
      │  sign JWT with private key
      ▼
AI Client / automation script
      │  Authorization: Bearer <jwt>
      ▼
MCPCore
      │  verify signature with your public key
      ▼
Tool executes
```

---

## When to use it

- **Server-to-server integrations** — your backend generates tokens for each call
- **Short-lived credentials** — tokens expire quickly, reducing risk if intercepted
- **Automated pipelines** — CI/CD, cron jobs, or internal services calling MCP tools programmatically
- **Multi-tenant systems** — embed per-tenant claims in the JWT payload

## Configure

<Steps>
  <Step title="Generate an RSA or EC key pair">
    Generate a key pair on your server. MCPCore supports RSA-256 and EC (ES256).

    ```bash
    # RSA 2048-bit (recommended)
    openssl genrsa -out private.pem 2048
    openssl rsa -in private.pem -pubout -out public.pem
    ```

    Keep `private.pem` secret on your server. You will give MCPCore the public key.
  </Step>
  <Step title="Open Server Settings">
    Go to your server's **Settings** tab in the MCPCore dashboard.
  </Step>
  <Step title="Select Bearer Token">
    Under **Security mode**, select **Bearer Token**.
  </Step>
  <Step title="Paste your public key">
    Copy the contents of `public.pem` and paste it into the **Public Key** field.

    <Frame>
      <img src="/images/screenshots/security-bearer-settings.png" alt="Bearer Token settings with public key field" />
    </Frame>
  </Step>
  <Step title="Save">
    Click **Save**. MCPCore will now validate the JWT signature on every request.
  </Step>
</Steps>

## Issuing a token from your backend

```javascript
import jwt from "jsonwebtoken";
import fs from "fs";

const privateKey = fs.readFileSync("./private.pem");

const token = jwt.sign(
  {
    sub:  "automation-pipeline",
    iss:  "your-app.example.com",
    aud:  "mcpcore",
    // Add any custom claims you need in your tool code
    tenantId: "acme-corp",
  },
  privateKey,
  {
    algorithm: "RS256",
    expiresIn:  "15m",      // short expiry — generate a new token per session
  }
);

// Pass `token` to your MCP client configuration
```

## Reading JWT claims in tools

Claims embedded in the JWT are available in your tool code:

```javascript
// Access standard claims
const subject  = env.JWT_SUB;   // the `sub` claim
const issuer   = env.JWT_ISS;   // the `iss` claim

// Access custom claims
const tenantId = env.JWT_TENANT_ID;

// Use them in your logic
const res = await sdk.http({
  method: "GET",
  url: `https://api.yourapp.com/tenants/${tenantId}/data`,
  headers: {
    Authorization: `Bearer ${env.INTERNAL_API_KEY}`,
  },
});

return JSON.parse(res.body());
```

<Note>
  MCPCore injects verified JWT claims as `env.JWT_*` variables. Only claims from a valid, unexpired token are injected.
</Note>

## Token expiry and renewal

JWTs have a built-in `exp` claim. MCPCore rejects expired tokens with `401 Unauthorized`. Your client should:

1. Check token expiry before each request
2. Generate a new token from your backend if the current one has expired (or is within 60 seconds of expiry)
3. Retry the request with the new token
